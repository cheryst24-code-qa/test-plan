# CONTRIBUTING.md
Данный документ описывает стандарты оформления баг-репортов и правила работы с задачами.
Это справочник для всей команды, чтобы дефекты оформлялись единообразно.

## Как оформить баг-репорт
Мы используем `[Название трекера: Jira / YouTrack / Trello]`. Перед созданием проверьте, нет ли дубликата в поиске.

### Обязательные поля
| Поле | Требование | Пример |
|------|------------|--------|
| **Title** | `[Module] Краткое описание проблемы` | `[Auth] Ошибка 500 при входе с OAuth` |
| **Environment** | ОС, браузер/устройство, версия сборки, env | `macOS 14.2, Chrome 124, Build v2.4.1-staging` |
| **Steps to Reproduce** | Нумерованный список, без лишних действий | `1. Открыть /login → 2. Нажать "Google" → 3. Выбрать аккаунт` |
| **Expected Result** | Что должно произойти по требованиям | `Успешный редирект на /dashboard` |
| **Actual Result** | Что произошло фактически + скриншот/лог | `Белый экран, в консоли: 500 Internal Server Error` |
| **Severity / Priority** | По матрице ниже | `Severity: High, Priority: Medium` |
| **Attachments** | Логи, скриншоты, видео, HAR-файл | `console.log, screenshot.png` |

### Матрица Severity vs Priority
| Severity (Влияние) | Priority (Срочность) | Когда применять |
|--------------------|----------------------|-----------------|
| **Critical** – потеря данных, падение сервиса, блокировка основного потока | **Immediate** – фикс в текущем спринте / срочно | Блокирует тестирование или работу пользователей |
| **High** – ключевая функция работает некорректно, обходной путь сложен | **High** – плановый фикс до релиза | Влияет на основной пользовательский сценарий |
| **Medium** – отклонение от требований, обходной путь существует | **Medium** – по capacity спринта | Не критично, но требует исправления |
| **Low** – косметика, опечатки, мелкие UX-неточности | **Low** – бэклог, рефакторинг | Не влияет на функциональность |

## Процесс работы
1. **QA**: создаёт задачу → назначает ответственного разработчика → вешает лейблы `bug`, `severity/...`, `priority/...`
2. **Dev**: переводит в `In Progress` → фиксирует → переводит в `Ready for QA`
3. **QA**: проверяет на тестовом окружении → если OK → `Verified` → `Closed`. Если NOK → `Reopened` с комментарием

## Советы по оформлению
- **Не описывайте гипотезы** в шагах воспроизведения. Только факты.
- **Если проблема плавающая**, добавьте `Воспроизводимость: 3/10 попыток` и приложите логи с таймстампами.
- **Прикрепляйте минимальный набор** данных для воспроизведения — это ускорит фикс.

> **Правило одного бага**: Одна задача = одна проблема. Не объединяйте несколько дефектов в одном тикете.

## Как оформить баг-репорт
Создавайте Issues только после проверки на `staging.taskflow.app` и поиска дублей.

### Обязательные поля
| Поле | Требование | Пример |
|------|------------|--------|
| **Title** | `[Module] Краткое описание проблемы` | `[Attachments] Загрузка PDF 51MB блокирует UI без сообщения об ошибке` |
| **Environment** | ОС, браузер, версия сборки, env | `macOS 14.4, Chrome 124.0.6367, Build v2.0.0-rc1, Staging` |
| **Steps to Reproduce** | Нумерованный список, без лишних действий | `1. Открыть задачу TF-405<br>2. Нажать "Upload"<br>3. Выбрать файл test_51mb.pdf` |
| **Expected Result** | Что должно произойти по требованиям | `Отображение ошибки "Максимальный размер 50MB", UI остаётся активным` |
| **Actual Result** | Что произошло фактически + скриншот/лог | `Белый экран на 15 сек, кнопка Upload недоступна, в консоли: `Error: PayloadTooLarge`` |
| **Severity / Priority** | По матрице ниже | `Severity: High, Priority: Medium` |
| **Attachments** | Логи, HAR-файл, видео, скриншоты | `console.log, network_har_51mb.zip` |

### Матрица Severity vs Priority
| Severity | Priority | Когда применять |
|----------|----------|-----------------|
| **Critical** | Immediate | Потеря данных, падение сервиса, блокировка основного потока (создание/редактирование задачи) |
| **High** | High | Ключевая функция работает некорректно, обходной путь сложен (файлы не скачиваются) |
| **Medium** | Medium | Отклонение от требований, обходной путь существует (некорректное превью JPG) |
| **Low/Cosmetic** | Low | Косметика, UX-неточности, опечатки в тултипах |

## Процесс работы
1. **QA**: создаёт Issue → назначает `Assignee: @frontend-dev` → вешает лейблы `bug`, `severity/high`, `priority/medium`
2. **Dev**: переводит в `In Progress` → фиксирует → переводит в `Ready for QA`
3. **QA**: проверяет на staging → если OK → `Verified` → `Closed`. Если NOK → `Reopened` с комментарием и новой сборкой.

## Шаблон Issue
Файл `.github/ISSUE_TEMPLATE/bug_report.md` уже содержит эту структуру. При создании Issue GitHub подставит её автоматически.
